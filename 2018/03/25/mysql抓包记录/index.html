记一次mysql协议抓包分析

发表于 2018-03-25 | 分类于 mysql
问题

开发同学自测出现问题，mysql proxy解析 mysql server response，无法正确解析datetime类型的数据，邀请一起研究下mysql 的date类型的数据结构，于是出现了如下事情：

果断打开浏览器，官方文档搜索，进入了如下页面二进制协议值
找到文档中关于date及datetime类型的协议包

ProtocolBinary::MYSQL_TYPE_DATE, ProtocolBinary::MYSQL_TYPE_DATETIME, ProtocolBinary::MYSQL_TYPE_TIMESTAMP:
如果年月日时分秒和毫秒都为空的话，编码长度为0
如果时、分、秒和毫秒都是0, 则编码的长度是4,为0部分则不发送
如果毫秒为0，长度为7，毫秒部分不发送
其他情况下，长度为11

  • length(1) –后续的字节长度(有效数据为0,4,7,11)
  • year(2) –year
  • month(1) –month
  • day(1) –day
  • hour(1)–hour
  • minite(1) –minutes
  • second(1) -seconds
  • micro_second(4) –micro-seconds
    例如：

第一个字节是04，代表后面有4位字节，代表其应为date类型，转化为标准时间为 2010-10-17

自觉已经明白了协议规范，边跟开发同学沟通，开发同学反馈，proxy自测接受到的报文，解析第一个字段是5位的，不符合0，4，7，11这一规范。
觉得一起先用mysql-cli和mysql-server通信来还原协议的解析，再debug proxy的报文细节，来定位问题。
mysql-cli执行sql，使用wireshark开始抓包，过程简单记录如下：

报文拿来分析，time类型的字段返回为
0a 12 33 ec ed dd

定位原因：不熟悉mysql协议，遗漏了关键信息null-bit类型和包头的pack[00].
